Baby Info — cum circulă o cerere prin aplicație

Ghid de studiu pentru modulul de Autentificare. Pentru fiecare operațiune urmărim drumul complet: din ce fișier frontend pleacă, prin ce fișiere backend trece, unde atinge baza de date și cum se întoarce răspunsul în pagină.

Cod de culori folosit peste tot:
FRONTEND pagina HTML + JavaScript din public/ CORE infrastructura prin care trece orice cerere (api/core/) CONTROLLER / MODEL logica modulului (api/controllers/, api/models/) BAZĂ DE DATE PostgreSQL (tabelele)

0. Imaginea de ansamblu — drumul universal al unei cereri

Indiferent ce faci (login, salvat copil, upload poză), fiecare cerere către server trece prin exact aceleași 5 etape. Învață odată acest schelet și restul documentului devine doar „ce se întâmplă în pasul 4".

[ Browser ] fetch('/api/auth/login', ...) # pagina ta HTML din public/ │ ▼ router.php # intră TOATE cererile. /api/* le trimite mai departe │ ▼ api/index.php # creează new App() și apelează run() │ ▼ api/core/App.php # pornește sesiunea + conexiunea la DB, apoi Router │ ▼ api/core/Router.php # potrivește URL-ul cu o rută → alege controllerul + metoda │ ▼ AuthController->login() # LOGICA propriu-zisă (pasul care diferă de la caz la caz) │ # ↕ vorbește cu UserModel → PostgreSQL ▼ api/core/Response.php # transformă rezultatul în JSON și îl trimite înapoi │ ▼ [ Browser ] res.json() → window.location = '/dashboard'
Ideea cheie: partea CORE (router → index → App → Router → Response) e mereu aceeași. Doar controllerul + modelul se schimbă în funcție de ce ceri. Când vrei să înțelegi o funcție nouă, sari direct la controllerul ei.

1. Cine sunt actorii (ce face fiecare fișier)

Stratul CORE — prin el trece orice

FișierRol
router.phpPunctul de intrare al serverului de dezvoltare. Decide: cerere către /api/... → trimite în api/index.php; fișier /uploads/... → serve_media.php; .css/.js → îl servește din public/; URL „curat" (ex. /login) → trimite pagina HTML corespunzătoare.
api/index.php3 linii: încarcă autoloaderul Composer, creează new App(), apelează run().
api/core/App.phpÎn constructor: setează fusul orar, pornește sesiunea PHP (cookie HttpOnly, SameSite=Strict) și deschide conexiunea PDO. run() creează Router-ul și apelează dispatch().
api/core/Router.phpAre o listă cu toate rutele (registerRoutes()). dispatch() compară URL-ul cerut cu fiecare rută, extrage parametrii (ex. {id}) și apelează metoda potrivită din controller. Tot aici se verifică limita de upload și se rezolvă Bearer token-ul (pentru API v1).
api/core/Request.phpAmbalează cererea HTTP: metoda, URL-ul, headerele, $_GET, $_FILES și body-ul (dacă e JSON îl decodează, altfel ia $_POST).
api/core/Response.phpTrimite răspunsul: Response::json($data, $status) setează codul HTTP, headerul JSON, scrie răspunsul și oprește execuția. Are și error(), xml(), html().
api/core/Controller.phpClasa-părinte a tuturor controllerelor. Oferă „gardienii": requireAuth(), requireCsrf(), requireWritePermission(), requireOwner(), requireFamilyAccess().
api/core/Security.phpUtilitare de securitate: generare/validare token CSRF, hash parolă (password_hash bcrypt), verificare parolă, escape XSS, generare token invitație/share.
api/core/SessionManager.phpÎnvelește $_SESSION. regenerate() schimbă ID-ul sesiunii la login; userId(), isAuthenticated(), isSuperAdmin() răspund „cine e logat acum?".
api/core/RateLimiter.phpAnti-brute-force: numără încercările eșuate (tabela auth_attempts) pe email + IP. Peste 5 eșecuri / 15 min → blochează cu cod 429.

Stratul modulului de Auth

FișierRol
api/controllers/AuthController.phpCreierul autentificării: metodele csrf, login, register, reset, logout, me.
api/models/UserModel.phpSingurul fișier care vorbește cu tabela users: findByEmail, findById, create, emailExists, updatePassword, verifySecurityAnswers etc. Toate cu prepared statements.
api/controllers/InviteController.phpGenerează linkul de invitație (store) și validează un token (validate).
api/models/InviteModel.phpTabela invitations: create, findByToken, markUsed.
api/models/FamilyModel.phpTabela family_members: leagă userul de copii cu o permisiune (owner/coparent/caregiver/viewer).
api/config/database.phpConexiunea PDO la PostgreSQL (singleton). Citește credențialele din .env.
api/config/constants.phpConstante: durata sesiunii, lungimi de token, ore de expirare invitație etc.

2. Token CSRF — apare înainte de orice formular

Înainte ca login/register/reset să trimită datele, JS-ul cere întâi un token CSRF. E un mecanism „double-submit cookie": serverul pune token-ul și într-un cookie, și în răspunsul JSON; la trimitere, browserul atașează cookie-ul automat, iar JS-ul atașează același token în headerul X-CSRF-Token. Serverul verifică că cele două coincid → dovadă că cererea vine din pagina ta, nu de pe alt site.

1
FRONTEND login.html (funcția getCsrfToken()) face fetch('/api/auth/csrf', {credentials:'include'}).
2
CORE router.php vede prefixul /api/ → trece în api/index.php → App.php → Router.php. Ruta GET /api/auth/csrf e mapată la AuthController::csrf.
3
CONTROLLER AuthController::csrf() apelează Security::generateCsrfToken().
4
CORE Security.php generează un token hex aleator (dacă nu există deja în cookie), îl pune în cookie-ul csrf_token (SameSite=Strict) și îl întoarce.
5
CONTROLLER răspunde cu Response::json(['token' => ...]). JS-ul îl ține într-o variabilă _csrfToken și îl refolosește la următoarele cereri din aceeași pagină.
Toate cele 3 helper-e getCsrfToken() din login/register/join/forgotpassword sunt identice. CSRF se cere o singură dată per pagină, apoi e cache-uit.

3. Login

Fișiere implicate: login.html → router.php → index.php → App.php → Router.php → AuthController.php → RateLimiter.php → UserModel.php → Security.php → SessionManager.php → Response.php

login.html (submit) └─ fetch POST /api/auth/login { email, password } + header X-CSRF-Token └─ router.php → index.php → App.php → Router.php └─ AuthController::login() ├─ requireCsrf() # verifică token CSRF ├─ RateLimiter::check('login') # max 5 eșecuri/15min? ├─ UserModel::findByEmail() ──► SELECT din users ├─ Security::verifyPassword() # compară cu hash bcrypt ├─ SessionManager::regenerate() # ID sesiune nou ├─ SessionManager::set('user_id') # te ține minte └─ Response::json(date user) └─ login.html: window.location = '/dashboard'

Pas cu pas

1
FRONTEND În login.html, la submit pe #loginForm: JS-ul face e.preventDefault() (oprește reîncărcarea paginii), citește email și password, schimbă butonul în starea „...WAIT", apoi face fetch('/api/auth/login', {method:'POST', headers:{Content-Type:json, X-CSRF-Token}, credentials:'include', body: JSON.stringify({email,password})}).
2
CORE router.php → api/index.php → App.php pornește sesiunea și DB → Router::dispatch() potrivește POST /api/auth/login și apelează AuthController::login().
3
CONTROLLER AuthController::login() apelează întâi requireCsrf() (din Controller.php), care compară headerul X-CSRF-Token cu cookie-ul prin Security::validateCsrfToken(). Token greșit → 403 și gata.
4
CONTROLLER Validează că email și password nu sunt goale (altfel 400). Apoi RateLimiter::check('login', $email).
5
DB RateLimiter.php face SELECT COUNT(*) din auth_attempts pentru acest email/IP în ultimele 15 min. Dacă ≥ 5 → 429 Too many attempts și se oprește aici.
6
MODEL UserModel::findByEmail() face un SELECT cu prepared statement din tabela DB users și întoarce rândul (sau null).
7
CONTROLLER Verificări în ordine: dacă userul nu există → recordFailure + 401 Invalid credentials; dacă banned_at nu e null → 403 Account suspended; Security::verifyPassword($password, $hash) greșit → recordFailure + 401.
8
CONTROLLER Succes: RateLimiter::clear('login') (șterge eșecurile), SessionManager::regenerate() (ID de sesiune nou — anti session-fixation), apoi salvează în sesiune user_id, first_name, is_superadmin, theme.
9
CONTROLLER Response::json([...datele userului...]) → cod 200, headerul de sesiune (cookie) merge înapoi la browser.
10
FRONTEND JS-ul verifică res.ok. Succes → window.location.href = '/dashboard'. Eroare → afișează data.error în caseta vintage și reactivează butonul.
Unde se „întâmplă" acțiunea: verificarea parolei și crearea sesiunii se petrec exclusiv în AuthController::login() (server). Frontend-ul nu validează parola — doar trimite datele și reacționează la răspuns. Asta e regula: autentificarea reală se face mereu pe server.

4. Register (cont nou)

Fișiere: register.html → CORE → AuthController.php → InviteModel.php (opțional) → UserModel.php → FamilyModel.php (opțional) → Response.php

register.html (submit) ├─ împarte „Full Name" în first_name + last_name # în JS ├─ verifică parola ≥ 8 caractere # validare rapidă în UI └─ fetch POST /api/auth/register { first_name, last_name, email, password, 3×security_answer } └─ AuthController::register() ├─ requireCsrf() ├─ (dacă există invite_token) InviteModel::findByToken() ──► SELECT invitations ├─ validează câmpuri + parolă ≥ 8 + cele 3 răspunsuri ├─ UserModel::emailExists() ──► SELECT users # 409 dacă există ├─ UserModel::create() ──► INSERT users # parolă + răspunsuri hash-uite └─ Response::json({id, message}, 201) └─ window.location = '/login?registered=1'

Pas cu pas

1
FRONTEND register.html: la submit, JS-ul împarte câmpul „Full Name" în first_name (primul cuvânt) și last_name (restul, sau "." dacă lipsește), verifică parola ≥ 8 caractere, apoi fetch POST /api/auth/register cu toate câmpurile + cele 3 răspunsuri de securitate.
2
CORE Același traseu router → index → App → Router. Ruta POST /api/auth/register → AuthController::register().
3
CONTROLLER requireCsrf(). Dacă body conține invite_token, validează invitația acum (vezi secțiunea Join). Apoi verifică: toate câmpurile prezente (400), parola ≥ 8 (400), cele 3 răspunsuri ne-goale (400).
4
MODEL UserModel::emailExists() → DB SELECT din users. Email deja folosit → 409 Email already registered.
5
MODEL UserModel::create() → DB INSERT în users. Aici se hash-uiesc parola (Security::hashPassword = bcrypt) și fiecare răspuns de securitate (lowercase + bcrypt). Returnează id-ul noului user.
6
CONTROLLER (Dacă era invitație) adaugă userul în familie prin FamilyModel și marchează invitația folosită — detaliat la Join. Apoi Response::json(['id'=>..., 'message'=>'Account created'], 201).
7
FRONTEND Succes → window.location = '/login?registered=1'. Pe pagina de login, parametrul registered=1 declanșează bannerul verde „Account created! Please log in." Notează: register nu te loghează automat (spre deosebire de Join).

5. Join — înregistrare prin invitație

Join e register cu doi pași în plus: (a) la încărcarea paginii validează tokenul ca să-ți arate la ce copil ești invitat, (b) după înregistrare te loghează automat. Folosește aceeași rută /api/auth/register, dar cu invite_token în body.

Fișiere: join.html → InviteController.php (validate) + AuthController.php (register) → InviteModel.php + UserModel.php + FamilyModel.php

Partea A — validarea tokenului (la încărcarea paginii)

1
FRONTEND join.html rulează validateToken() automat la final. Citește ?token= din URL și face fetch('/api/invite?token=...'). Cât timp așteaptă, arată skeletonul de încărcare.
2
CONTROLLER Ruta GET /api/invite → InviteController::validate().
3
MODEL InviteModel::findByToken() → DB SELECT din invitations JOIN children (ca să ia și numele copilului), doar dacă used_at IS NULL. Token inexistent → 404; expirat (expires_at < acum) → 410.
4
CONTROLLER Răspunde cu child_id, child_name, permission, expires_at.
5
FRONTEND Ascunde skeletonul, arată bannerul cu numele copilului și rolul (Owner/Co-Parent/Caregiver/Viewer) și afișează formularul. Token lipsă/invalid → arată ecranul „Invitation Not Found".

Partea B — înregistrarea + auto-login

6
FRONTEND La submit: fetch POST /api/auth/register cu toate câmpurile + invite_token.
7
CONTROLLER AuthController::register() revalidează tokenul (InviteModel::findByToken; expirat → 410), creează userul (UserModel::create), apoi: ia toți copiii invitatorului prin FamilyModel::getChildIdsByUser() și pentru fiecare face FamilyModel::addMember(childId, userNou, permission) → DB INSERT în family_members. La final InviteModel::markUsed() → DB UPDATE invitations (pune used_at, used_by) ca tokenul să nu mai poată fi refolosit.
8
FRONTEND După 201, join.html face imediat un al doilea fetch POST /api/auth/login cu același email/parolă. Reușit → /dashboard; eșuat → /login?registered=1.
De ce link-ul de invitație se generează separat? În InviteController::store() (POST /api/children/{id}/invites), apelat din dashboard de către owner. Acolo requireOwner() verifică drepturile, Security::generateInviteToken() creează tokenul, iar InviteModel::create() îl salvează în invitations cu o expirare (72h). Linkul rezultat e cel pe care îl deschide noul membru în join.html.

6. Forgot password (resetare cu întrebări de securitate)

Nu există email de recuperare — resetarea se face într-un singur pas, verificând cele 3 răspunsuri de securitate setate la înregistrare.

Fișiere: forgotpassword.html → AuthController.php → RateLimiter.php → UserModel.php → Response.php

forgotpassword.html (submit) └─ fetch POST /api/auth/reset { email, 3×security_answer, new_password } └─ AuthController::reset() ├─ requireCsrf() ├─ validează email + 3 răspunsuri + parolă ≥ 8 ├─ RateLimiter::check('reset') ├─ UserModel::findByEmail() ──► SELECT users ├─ UserModel::verifySecurityAnswers() ──► SELECT users # compară hash-uri │ └─ greșit → recordFailure + 400 (mesaj generic) ├─ UserModel::updatePassword() ──► UPDATE users # hash nou └─ Response::json({message}) └─ arată confirmarea → redirect /login după 1.5s

Pas cu pas

1
FRONTEND forgotpassword.html: strânge email, cele 3 răspunsuri și parola nouă; verifică ≥ 8 caractere; fetch POST /api/auth/reset cu header CSRF.
2
CONTROLLER AuthController::reset(): requireCsrf(), normalizează emailul (lowercase+trim), verifică răspunsurile ne-goale și parola ≥ 8 (altfel 400).
3
DB RateLimiter::check('reset', email) — răspunsurile au entropie mică, deci se limitează ghicirea (5/15min → 429).
4
MODEL UserModel::findByEmail() + verifySecurityAnswers() → două SELECT-uri din DB users. verifySecurityAnswers compară fiecare răspuns dat (lowercase) cu hash-ul din DB prin password_verify. Orice nepotrivire → false.
5
CONTROLLER Email greșit SAU răspunsuri greșite → același mesaj generic 400 "Email or security answers are incorrect" (nu dezvăluie care dintre ele) + recordFailure. Corect → RateLimiter::clear.
6
MODEL UserModel::updatePassword() → DB UPDATE users cu noul hash bcrypt. Apoi Response::json(['message'=>'Password has been reset']).
7
FRONTEND Ascunde formularul, arată mesajul „Password reset!", apoi setTimeout → /login după 1.5s.
PasswordResetModel.php (cu coduri temporare în tabela password_resets) e o variantă alternativă scrisă de coleg; flow-ul tău activ folosește întrebările de securitate prin UserModel.

7. „Cine sunt eu?" (me) și Logout

me e gardianul folosit de paginile protejate (ex. dashboard): la încărcare întreabă serverul dacă mai ești logat.

1
FRONTEND O pagină protejată face fetch('/api/auth/me', {credentials:'include'}). Cookie-ul de sesiune merge automat.
2
CONTROLLER AuthController::me() apelează requireAuth(). Dacă SessionManager::isAuthenticated() e fals → 401 (și frontendul te redirectează la /login).
3
MODEL Logat → UserModel::findById() → DB SELECT users, răspuns cu datele tale (id, nume, rol, temă...).
4
CONTROLLER Logout: POST /api/auth/logout → AuthController::logout() → SessionManager::destroy() șterge $_SESSION și cookie-ul → 200. Frontendul te duce la login.

Asta e tot.

9. Dashboard copii (CRUD + poză + feeding/sleep/growth + invite)

Dashboard-ul e prima pagină după login. La încărcare verifică sesiunea, încarcă lista de copii și, pentru fiecare, ultimele evenimente. De aici se fac aproape toate acțiunile de „scriere" pe un copil.

Fișiere: dashboard.html → CORE → ChildController.php / FeedingController.php / SleepController.php / GrowthController.php / InviteController.php → modelele lor → UploadService.php (poză) → Response.php

9.1 Încărcarea paginii (guard + listare)

1
FRONTEND dashboard.html cere întâi token CSRF, apoi fetch('/api/auth/me'). Răspuns 401 → window.location='/login' (guard-ul de pagină).
2
FRONTEND fetch('/api/children') → CONTROLLER ChildController::index() → ChildModel::getByUser() → DB SELECT din children JOIN family_members (aduce și coloana permission a userului). Din această permisiune decide UI-ul ce butoane arată (owner vede „Invite", caregiver/viewer e redirectat la gallery).
3
FRONTEND Pentru fiecare copil face în paralel /feedings?limit=3, /sleep?limit=3, /growth?limit=1 ca să afișeze rezumatul. Fiecare merge la controllerul corespunzător → index() → model → SELECT.

9.2 Adăugare / editare copil

dashboard.html: formular „Add child" (submit) └─ fetch POST /api/children { first_name, date_of_birth, gender, ... } + X-CSRF-Token └─ ChildController::store() ├─ requireAuth() + requireCsrf() ├─ validează first_name + date_of_birth obligatorii ├─ ChildModel::create() ──► INSERT children └─ ChildModel::addFamilyMember() ──► INSERT family_members (owner) └─ Response::json({id}, 201) └─ reîncarcă lista de copii
1
CONTROLLER ChildController::store() cere requireAuth() + requireCsrf(), validează câmpurile, apoi ChildModel::create() (DB INSERT children) și imediat addFamilyMember(child, user, 'owner') — adică cel care creează copilul devine automat owner-ul lui.
2
CONTROLLER Editarea (PUT /api/children/{id} → update()) folosește requireWritePermission() (doar owner/coparent), revalidează aceleași câmpuri și un gen din whitelist M/F/other, apoi ChildModel::update() → UPDATE children. Ștergerea (DELETE → destroy()) → ChildModel::delete() → DELETE children → 204 No Content.

9.3 Upload poză de profil (multipart + finfo)

Singura cerere din auth/dashboard care NU e JSON, ci FormData (fișier binar). Fișierul se salvează în afara folderului public.

dashboard.html: input file 'photo' → FormData └─ fetch POST /api/children/{id}/photo (multipart/form-data) + X-CSRF-Token └─ ChildController::uploadPhoto() ├─ requireWritePermission() ├─ finfo → MIME real ∈ {jpeg,png,webp}? # 422 dacă nu ├─ size < 5MB? ├─ move_uploaded_file → storage/uploads/photos/ # AFARĂ din webroot └─ ChildModel::updatePhoto() ──► UPDATE children.photo_url └─ Response::json({photo_url:'/uploads/photos/...'}) └─ img.src = photo_url
De ce poza nu e link direct? Fișierele stau în storage/uploads/, în afara folderului public/, deci browserul nu le poate citi direct. Când <img> cere /uploads/photos/x.jpg, router.php trimite cererea la api/serve_media.php, care verifică anti-path-traversal cu realpath, validează MIME-ul real și abia apoi trimite fișierul (cu suport HTTP Range pentru video/audio). E mecanismul cerut de curs: „store files outside webroot".

9.4 Logare feeding / sleep / growth

Aceeași schemă pentru toate trei, doar validările diferă (sincronizate cu constrângerile CHECK din DB ca să nu pice cu 500).

AcțiuneRutăController → Model → DBValidări cheie (server)
Log feedingPOST /api/children/{id}/feedingsFeedingController::store → FeedingModel::create → INSERT feedingstype ∈ breast/bottle/solids; side ∈ L/R/both; durată 0–600; cantitate 0–2000; data nu în viitor
Log sleepPOST /api/children/{id}/sleepSleepController::store → SleepModel::create → INSERT sleep_logsstarted_at obligatoriu; type ∈ night/nap; quality 1–5; ended_at > started_at; fără date în viitor
Log growthPOST /api/children/{id}/growthGrowthController::store → GrowthModel::create → INSERT growthcel puțin o măsurătoare; weight≤200, height≤250, head≤100; fără date în viitor
1
FRONTEND Modalul „ADD LOG" din dashboard.html trimite JSON la ruta potrivită cu header CSRF.
2
CONTROLLER Toate trei store(): requireAuth() + requireCsrf() + requireWritePermission(childId), apoi validările din tabel, apoi create() în model cu logged_by = SessionManager::userId() (reține cine a logat). Răspuns 201.
3
MODEL getByChild() (pentru index) face SELECT ... JOIN users ca să afișeze și numele celui care a logat evenimentul, ordonat descrescător după dată.

9.5 Invite din dashboard

1
FRONTEND Owner-ul deschide modalul Invite și trimite POST /api/children/{id}/invites cu email + permission.
2
CONTROLLER InviteController::store() → requireOwner() (doar owner invită) → Security::generateInviteToken() → InviteModel::create() → DB INSERT invitations cu expirare 72h. Răspunde cu linkul /invite?token=... pe care UI-ul îl afișează pentru copiere.
Mai departe, noul membru deschide linkul în join.html — vezi secțiunea 5. Aici se închide cercul invite → join.

10. Gallery / momente (timeline, media, reactions, comentarii, tags, filtre)

Gallery e „raftul" cu polaroide grupate pe luni. Un moment e o intrare din timeline (milestone, poză, notă) și poate avea media atașată, tag-uri, reacții și comentarii. Este și pagina la care ajung caregiver/viewer (read-only, doar momentele marcate is_shared).

Fișiere: gallery.html → CORE → TimelineController.php / MomentController.php / CommentController.php / ReactionController.php → MomentModel / MediaModel / CommentModel / ReactionModel → UploadService.php (media)

10.1 Afișarea timeline-ului

12. Cum rulează proiectul (server PHP vs Apache) — și de ce există .htaccess

Proiectul nu folosește XAMPP/Apache pentru rulare. Folosește serverul web încorporat în PHP plus PostgreSQL separat. Important: serverul încorporat al PHP-ului nu citește deloc fișierele .htaccess — acelea sunt doar pentru Apache.

12.1 Cum pornește efectiv

1
DB Pornește serviciul PostgreSQL (port 5433, baza babyinfo) cu schema din database/schema.sql importată.
2
CORE Din folderul proiectului rulezi comanda din start-server.bat:
php -S localhost:8000 -d upload_max_filesize=50M -d post_max_size=52M router.php
Asta lansează serverul web încorporat al PHP-ului pe localhost:8000, cu router.php drept front-controller.
3
FRONTEND Deschizi http://localhost:8000 în browser.
Limitele upload_max_filesize=50M / post_max_size=52M ridică valorile implicite din php.ini (2M), necesare pentru upload de video. Doar pentru sesiunea curentă.

12.2 Două moduri de rutare = portabilitate

Proiectul are două mecanisme care fac aceeași rutare, pentru două moduri de a rula. Niciodată amândouă în același timp.

MecanismFolosit când...Ce face
router.phprulezi cu serverul PHP încorporat (modul vostru de dezvoltare)front-controller: /api/*→api/index.php, /uploads/*→serve_media.php, URL curate→pagina HTML, restul→fișiere statice din public/
api/.htaccessproiectul rulează pe Apache (ex. XAMPP)trimite tot ce nu e fișier/folder real către api/index.php (echivalentul rutării API)
public/.htaccessproiectul rulează pe Apachemapează URL curate (/login→login.html etc.) și are fallback la landing page
De ce păstrăm .htaccess deși nu le folosim acum: sunt pentru portabilitate. Dacă proiectul e deschis pe Apache/XAMPP (de un coleg sau de profesor), aceste fișiere fac aplicația să meargă fără nicio modificare — Apache preia exact rolul pe care îl are router.php la noi. Cost zero, plus de compatibilitate. Conținutul lor trebuie ținut în sincron cu router.php: orice pagină nouă din pageMap trebuie să aibă și o regulă în public/.htaccess (ex. regula pentru /family a fost adăugată tocmai pentru a păstra această consistență).
